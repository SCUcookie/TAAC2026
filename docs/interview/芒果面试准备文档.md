  2. 核心面试问题与回答策略

  Q1: 你在简历中提到了“业务场景驱动的周时段权重分配”，具体是如何实现的？为什么这么做？

  面试官意图: 考察你对业务数据的敏感度以及特征工程的落地能力。

  你的回答逻辑:
   1. 业务洞察: 我们发现芒果TV的用户行为有明显的周周期性。
       * 周三: 往往是热门综艺（如《乘风破浪》等）更新的时间，用户目的性强，活跃度高。
       * 周五-周日: 休闲时间，用户探索意愿强，流量大。
       * 周一/周二: 工作日初期，活跃度相对较低。
   2. 代码实现 (基于 `CTR.py` 中的 `add_time_weight` 函数):
       * 我设计了一个时间衰减 + 周期加权的组合权重策略。
       * 基础衰减: 近期行为权重高，远期权重低。但我做了一个优化（alpha_dynamic），对于高活跃用户（平均交互间隔 <2 天），我让他们的权重衰减得更慢（alpha +
         0.05），因为他们的历史兴趣更能代表当前偏好；低活跃用户则衰减得快，更依赖极短期的行为。
       * 周期加权 (`day_of_week`):
           * 周末 (Sat/Sun): 给予 1.5倍 权重 (代码中 [0, 1] 对应的情况)。
           * 周初 (Mon/Tue): 给予 0.8倍 权重 (代码中 [2, 3])，降低噪声干扰。
           * 其他 (Wed/Thu/Fri): 保持 1.0 基准权重。
   3. 效果: 这种加权方式让模型在训练时更“关注”那些高质量的活跃时间段，在验证集上提升了AUC。

  代码对应:

   1 # 你的代码 CTR.py 中真实存在的逻辑
   2 df_hist = df_hist.with_columns(
   3     pl.when(pl.col('day_of_week').is_in([0, 1]))  # 周六日
   4     .then(1.5)
   5     .when(pl.col('day_of_week').is_in([2 ,3]))  # 周一二
   6     .then(0.8)
   7     .otherwise(1.0)
   8     .alias('weekend_weight')
   9 )

  Q2: 简历中提到了 MMoE (多任务混合专家模型)，但项目中主要使用了 CatBoost，这两者是如何结合的？

  面试官意图: 这是一个关键点。如果你的代码主要是CatBoost，你需要解释清楚架构，避免被认为“简历造假”或“概念混淆”。

  回答策略 (非常重要):
  你可以将你的系统描述为 "Deep & Shallow" (深层与浅层结合) 的两阶段架构，或者解释 MMoE 是你尝试的深度学习方案，而最终竞赛提交为了稳健性使用了 CatBoost 融合 MMoE 提取的特征。

  推荐回答:
  "在我的项目中，我设计了一个多阶段的预测系统：
   1. 底层表征学习 (Shared Bottom / Embedding Layer):
       * 我使用了 双塔 DNN (YoutubeDNN) (对应 youtubeDNN.py) 来学习用户(User)和视频(Video)的向量表示。这部分起到了 'Shared Bottom' 的作用，通过对比学习（正样本点击 vs 负样本曝光未点击）提取了高维的语义特征。
   2. 任务专家与门控 (MMoE 思想的体现):
       * 在深度学习实验版本中，我尝试了标准的 MMoE 结构来同时优化 CTR 和 CR 任务。
       * 但在最终的 CatBoost 模型中 (对应 M_CTR_2_0.ipynb Step 9)，我将这些 DNN 生成的 Embedding 作为重要特征输入。
       * 这就相当于：DNN 负责提取通用的、泛化的多模态特征（共享层），而 CatBoost 这一步充当了极强的 '门控' 和 '专家' 角色，利用树模型的特征分裂能力，针对 '点击' 和 '完播' 两个不同的 Label 分别进行精细化的拟合。
   3. 为什么这么做: 比赛数据通常是表格型的，Tree Model (CatBoost) 在处理统计特征（如滑窗统计、计数特征）上优势巨大，而 DNN 擅长处理 ID 类稀疏特征。将两者结合（DNN提取特征 -> CatBoost分类）通常能达到最佳效果。"

  Q3: 关于完播率 (Completion Rate) 预测，你是怎么处理的？(CR.py)

  面试官意图: 完播率是一个回归问题，不同于点击率的分类问题，考察你对回归任务的处理。

  你的回答:
   1. 目标定义: 完播率 = play_time / item_duration。
   2. 数据清洗: 代码中我注意到有些异常值（完播率 > 1），虽然 CR.py 主要是读取处理好的数据，但在逻辑上我会截断超过 100% 的数值，并将缺失值填充为 0。
   3. 模型选择: 使用了 CatBoostRegressor，损失函数选择了 RMSE (均方根误差)。
   4. 特征工程:
       * 除了常规的用户/视频特征，我特别引入了 弹幕密度特征 (get_df_danmu_features 函数)。
       * 逻辑: 弹幕多的视频通常互动性强，完播率往往更高（或者两极分化）。我计算了 danmu_count / item_duration 作为密度特征加入模型，这对预测用户是否会坚持看完很有帮助。

  Q4: 你是如何解决“冷启动”问题的？(代码中的 README_冷启动解决方案.md)

  面试官意图: 冷启动是推荐系统的经典难题。

  你的回答:
  我采用了一套分层处理策略：
   1. 识别: 首先将测试集用户分为 老用户 (Warm Start) 和 新用户 (Cold Start)。
   2. 老用户: 走主模型 (CatBoost)，利用丰富的历史行为特征进行精准预测。
   3. 新用户 (策略部分):
       * 由于没有历史点击，无法生成个性化特征。
       * 基于热度的召回: 我统计了全局的 热门视频 (Hot Videos)（基于 show_count 曝光量或 click_count 点击量）。
       * 默认值填充: 对于新用户的点击率/完播率预测，我给予了一个基于全局统计的置信度（如完播率预设为 0.4），保证他们能看到大盘最热的内容。
       * (进阶回答): 代码中还预留了用户聚类的接口，可以通过新用户的有限属性（如设备信息、来源渠道）匹配到相似的老用户群体，利用该群体的平均偏好进行推荐。

  Q5: 你的 Score 0.276666 是怎么算出来的？MMoE 的联合得分公式是什么？

  面试官意图: 考察对比赛指标的理解。

  你的回答:
  比赛的评估指标通常是 CTR 和 CR 的加权组合。
  在我的架构中：
   * Recall (召回) 阶段: 使用 DNN Embedding 相似度或规则召回。
   * Ranking (精排) 阶段:
       * 模型 A 预测 $P(Click)$ (点击概率)。
       * 模型 B 预测 $P(Completion)$ (完播率)。
   * 最终打分 (Fusion): $Score = P(Click) \times P(Completion)$ (或者是加权 $w_1 \times CTR + w_2 \times CR$)。
   * 代码 CR.py 最后的逻辑展示了这一点：

   1     user_test_pred_final = user_test_pred_final.with_columns(
   2         (pl.col("pred_completion") * pl.col("pred_label")).alias("completion_rate")
   3     )
      我最终是取两者的乘积作为排序依据。这符合业务逻辑：我们希望推荐的是“用户既想点进去，又愿意看完”的高质量视频。
